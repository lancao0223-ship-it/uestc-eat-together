# 成电约饭 — 产品需求文档（PRD）

> **产品名称**：成电约饭  
> **目标平台**：微信小程序（优先）+ iOS/Android App（二期）  
> **目标用户**：电子科技大学清水河校区 & 沙河校区在校生  
> **文档版本**：v1.0  
> **日期**：2026-04-26

---

## 1. 问题陈述（Problem Statement）

电子科技大学两校区在校生超过 4 万人，但校园生活圈相对封闭，学生经常面临「一个人吃饭无聊、想探店却缺饭搭子、外卖凑不够起送价」等问题。现有的通用社交平台（微信、QQ、小红书）缺乏针对校园场景的约饭撮合机制，而表白墙/闲置群的信息又过于杂乱、实时性差。如果缺乏一个专注校园场景、轻社交重实用的约饭工具，学生的日常用餐体验将继续被这些碎片化问题困扰。

---

## 2. 目标（Goals）

| 目标 | 类型 | 说明 |
|------|------|------|
| **G1. 月活跃用户数（MAU）达到在校生的 20%** | 用户目标 | 上线 3 个月内，MAU ≥ 8,000 人，证明场景需求真实存在 |
| **G2. 人均每周成功约饭 ≥ 1.5 次** | 用户目标 | 证明产品不仅被打开，还能真正促成用餐撮合 |
| **G3. 约饭爽约率 ≤ 5%** | 用户目标 | 通过信用体系约束，建立可信赖的社区氛围 |
| **G4. 小程序次日留存率 ≥ 35%** | 业务目标 | 证明产品在用户心智中形成习惯，而非一次性尝鲜 |

---

## 3. 非目标（Non-Goals）

| 非目标 | 理由 |
|--------|------|
| **NG1. 不做陌生人交友/恋爱匹配功能** | 避免产品被贴上「 dating app 」标签，保持「约饭工具」的心智定位。社交属性通过「口味相投」自然发生，不做主动撮合 |
| **NG2. 不做外卖拼单的全流程交易** | 外卖拼单仅做「凑单撮合」，支付环节仍跳转美团/饿了么，避免涉及资金托管和餐饮合规风险 |
| **NG3. 不做校外商家团购/优惠券** | 商业化方向暂不考虑，专注核心撮合体验。团购属于独立业务线，后续可探索但 v1 不碰 |
| **NG4. 不做跨校区约饭（清水河 ↔ 沙河）** | 两校区物理距离 15km+，实际约饭可行性极低。v1 限定同校区，降低撮合失败率 |
| **NG5. 不做完整的即时通讯（IM）** | 约饭确认后提供临时群聊/留言板即可，深度聊天引导至微信。自建 IM 成本极高且与微信生态冲突 |

## 4. 用户故事（User Stories）

### 4.1  persona：想找人一起吃饭的普通学生（小张）

- **US-1**：作为想找人一起吃饭的学生，我想要发起一个「今天中午 12:00 南门商业街轻食店」的约饭，并设定人数上限，以便快速找到饭搭子。
- **US-2**：作为想尝试商业街新开火锅店的学生，我想要发起「今晚 18:30 老码头火锅 3=1」的约饭，并标注 AA 制，以便找到口味相近且消费观一致的同伴。
- **US-3**：作为已经发起约饭的学生，我想要随时查看「已报名人员」的头像、性别和口味标签，以便决定是否确认成行。
- **US-4**：作为报名者，我想要在约饭前给发起人发送一条简短的申请留言（如「我不吃香菜」），以便提前沟通饮食禁忌。
- **US-5**：作为约了饭的学生，我想要在约饭结束后给同行人打标签（守时/好聊/推荐），以便积累社交信用。

### 4.2 persona：外卖凑不够起送价的学生（小王）

- **US-6**：作为想点奶茶但起送价 20 元我只买一杯 12 元的学生，我想要发布「外卖凑单」请求（商家、目标金额、当前缺口、配送地址），以便找到同宿舍楼拼单的人。
- **US-7**：作为看到凑单请求的学生，我想要一键「加入凑单」并查看已有订单明细，以便确认拼单内容后一起下单。

### 4.3 persona：注重安全与体验的学生（小陈）

- **US-8**：作为女生，我想要在发起/加入约饭时选择「仅女生可见」或「需实名认证」，以便提升安全感。
- **US-9**：作为曾经遇到爽约情况的学生，我想要查看对方的「信用分」和「历史爽约次数」，以便判断是否值得约饭。
- **US-10**：作为注重饮食健康安全的学生，我想要在个人主页上传体检报告（如乙肝五项等），并在加入约饭时选择「仅允许上传体检报告的用户加入」，以便降低聚餐时的健康风险。

## 5. 需求（Requirements）

### 5.1 Must-Have（P0）—— 没有这些不能上线

#### R1. 约饭发布与加入
- **描述**：用户可以发起约饭，填写时间、地点（商家名或具体位置）、人数上限、消费方式（AA / 我请 / 求请）、口味标签（辣度、忌口、偏好）。其他用户可以浏览并一键申请加入。
- **验收标准**：
  - [ ] 地点支持手动输入或从历史记录选择，不做校内/校外分类限制
  - [ ] 约饭卡片在首页按「时间最近」排序，距离开饭 > 2 小时的帖子自动沉底
  - [ ] 发起者可以「确认成行」或「取消约饭」，确认后生成临时约饭详情页（含参与者微信二维码/群聊入口）
  - [ ] 单个约饭人数达到上限后自动关闭报名，已报名用户收到通知

#### R2. 用户认证与基础画像
- **描述**：用户需通过校园邮箱（@uestc.edu.cn / @std.uestc.edu.cn）或学工认证完成身份验证，确保社区封闭性。
- **验收标准**：
  - [ ] 支持校园邮箱验证码认证，邮箱域名白名单可配置
  - [ ] 认证后自动生成基础画像：昵称、性别、校区、年级、口味标签（至少选 3 个）
  - [ ] 未认证用户只能浏览，不能发起或加入约饭

#### R3. 信用与爽约记录
- **描述**：系统记录每位用户的约饭履约情况，建立基础信用体系。
- **验收标准**：
  - [ ] 每次约饭结束后，参与者互相标记「已履约 / 爽约 / 未评价」，默认「已履约」
  - [ ] 被标记爽约的用户信用分 -10，初值为 100，低于 80 分时头像旁显示「低信用」标识
  - [ ] 爽约记录在个人主页可见，且发起约饭时他人可查看

#### R4. 首页 Feed 与筛选
- **描述**：首页展示当前校区内所有进行中的约饭/凑单请求，支持基础筛选。
- **验收标准**：
  - [ ] 默认按「距离我所在位置最近」排序（基于用户手动选择的常去区域，非 GPS 精确定位，保护隐私）
  - [ ] 支持按「类型（约饭/凑单）」和「时间段（现在/今天/明天）」筛选
  - [ ] 空态提示友好：「暂时没有约饭，去发起一个吧～」

---

### 5.2 Nice-to-Have（P1）—— 显著提升体验，但核心流程不依赖

#### R5. 口味标签智能匹配
- **描述**：系统根据用户填写的口味标签，在约饭卡片上显示「匹配度 85%」等提示，帮助用户快速找到口味相近的饭搭子。
- **验收标准**：
  - [ ] 匹配度算法基于标签重合度 + 历史评价中的口味关键词
  - [ ] 匹配度 ≥ 70% 时，在 Feed 中该卡片高亮显示

#### R6. 同楼层/附近实时提醒
- **描述**：用户可订阅「我所在的宿舍楼/教学楼附近」的约饭推送，减少主动刷新的成本。
- **验收标准**：
  - [ ] 用户可设置 1-3 个「关注区域」（如「品学楼 / 本科 24 栋 / 南门商业街」）
  - [ ] 当关注区域内有新约饭发布时，推送微信小程序订阅消息（非强制通知，每日上限 3 条）

#### R7. 外卖凑单撮合
- **描述**：辅助性的「外卖凑单」频道，仅支持同宿舍楼同学凑外卖起送价，降低因校区面积大导致的取货成本。
- **验收标准**：
  - [ ] 凑单发布需填写：商家名、目标金额、当前缺口、期望送达时间、取货地点（限定为宿舍楼，不可选其他位置）
  - [ ] 凑单仅对同宿舍楼用户可见，凑单成功后生成临时群聊，引导用户在美团/饿了么分别下单后统一取货

#### R8. 约饭临时群聊/留言板
- **描述**：约饭确认后，参与者进入一个 24 小时有效的临时留言板，用于沟通具体碰头位置、时间微调等。
- **验收标准**：
  - [ ] 留言板支持文字 + 图片，不支持语音/视频
  - [ ] 约饭时间到达后 2 小时，留言板自动归档只读

### 5.3 Future Considerations（P2）—— v1 不实现，但架构需预留

| 需求 | 说明 |
|------|------|
| **R10. 商家合作与优惠** | 与商业街/南门商家合作，约饭成功后发放优惠券。需预留商家后台接口 |
| **R11. 跨校区约饭（如科研合作）** | 研究生/博士生可能需要跨校区约饭，v1 不做但数据模型需支持校区字段扩展 |
| **R12. 饭局相册/探店评价沉淀** | 允许用户上传饭局照片、评价商家，形成 UGC 内容。需预留内容审核机制 |
| **R13. 积分商城** | 信用积分可兑换学校周边/食堂代金券。需预留积分流水表结构 |
| **R14. 班级/课题组圈子** | 支持小圈子私密约饭（如「我们实验室的人」）。需预留群组权限模型 |

## 6. 成功指标（Success Metrics）

### 6.1 先导指标（Leading Indicators）—— 上线后 1-4 周可见

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| **认证转化率** | ≥ 60%（打开小程序 → 完成校园认证） | 小程序埋点，漏斗分析 |
| **约饭发布量** | 日均 150+ 条（两校区合计） | 数据库统计 |
| **约饭成功率** | ≥ 70%（发起 → 确认成行） | 数据库统计，成行标记 |
| **Feed 刷新频次** | 人均每日 3+ 次 | 小程序埋点 |

### 6.2 滞后指标（Lagging Indicators）—— 上线后 1-3 个月可见

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| **月活跃用户数（MAU）** | ≥ 8,000（在校生 20%） | 小程序后台 + 自建用户表去重 |
| **约饭周频次** | 人均 ≥ 1.5 次 | 用户维度统计约饭成行次数 |
| **7 日留存率** | ≥ 35% | 小程序留存分析 |
| **爽约率** | ≤ 5% | 约饭评价中「爽约」标记占比 |
| **NPS 评分** | ≥ 40 | 每月抽样问卷（问卷星） |

## 7. 开放问题（Open Questions）

| 问题 | 需要回答的人 | 阻塞性 | 说明 |
|------|-------------|--------|------|
| **Q1. 是否需要接入学校统一身份认证系统？** | 工程 + 校方合作 | **阻塞** | 若学校不配合，退而求其次用校园邮箱认证 |
| **Q2. 校外商家数据由谁维护与更新？** | 运营 + 产品 | 非阻塞 | 商家开业/关闭/更名较频繁，需确定更新机制与信息来源 |
| **Q3. 女生安全功能是否需要接入学校保卫处？** | 产品 + 校方 | 非阻塞 | 若发生线下纠纷，产品免责边界需提前明确 |
| **Q4. 体检报告涉及健康隐私，如何确保数据安全与合规？** | 工程 + 法务 | **阻塞** | 需明确报告存储方式、查看权限、过期提醒机制，避免隐私泄露风险 |
| **Q5. 小程序订阅消息触达率较低，是否考虑企业微信/公众号辅助？** | 工程 + 运营 | 非阻塞 | 微信订阅消息需要用户主动授权，需设计合理的引导时机 |
| **Q6. 是否允许毕业校友使用？** | 产品 | 非阻塞 | 校友有聚餐需求，但会破坏「在校生封闭社区」的定位 |

## 8. 时间线考量（Timeline Considerations）

### 8.1 关键时间节点

- **开学季（9 月初）**：最佳冷启动时间，新生对社交需求最强，老生回归校园生活节奏。
- **期中考试后（10 月中下旬）**：若错过开学季，考试后学生社交意愿回升，可作为备选上线窗口。
- **双十一/双十二**：外卖凑单需求天然高峰期，若此时上线需确保 R7（外卖凑单）至少具备 MVP 能力。

### 8.2 建议分阶段交付

| 阶段 | 周期 | 交付内容 | 目标 |
|------|------|----------|------|
| **MVP** | 4-5 周 | R1-R4（约饭发布、认证、信用、首页 Feed） | 验证核心撮合需求，获取种子用户 1,000 人 |
| **v1.1** | 2-3 周 | R5-R6（口味匹配、附近提醒）+ 体验优化 | 提升匹配效率，MAU 冲 5,000 |
| **v1.2** | 2 周 | R7（外卖凑单）+ 分享裂变功能 | 利用外卖场景拉新，MAU 冲 8,000 |
| **v2.0** | 视数据而定 | P2 需求评估 + 商家合作探索 | 寻找商业模式 |

### 8.3 依赖项

1. **微信小程序认证与审核**：需提前注册企业/个体户主体，预计 1-2 周。
2. **校园邮箱验证服务**：若学校无统一 API，需自建邮箱验证码服务，依赖短信/邮件网关。
3. **服务器与域名**：建议部署在国内云（阿里云/腾讯云），需完成 ICP 备案（约 2-4 周）。

## 附录：关键术语表

| 术语 | 说明 |
|------|------|
| **约饭** | 用户发起的共同用餐邀约，含时间、地点、人数等信息 |
| **饭搭子** | 共同参与约饭的用户 |
| **信用分** | 基于履约记录的用户信誉评价体系，初值 100 |
| **成行** | 约饭发起人确认「人去、时间到、饭吃了」，标志着一次约饭成功完成 |
| **爽约** | 约饭确认后，某一方未按约定出现，被标记为爽约 |
